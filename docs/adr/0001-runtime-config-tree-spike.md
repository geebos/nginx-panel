# ADR 0001: Runtime configuration tree spike

- Status: Accepted
- Date: 2026-07-19
- Scope: TECHNICAL_DESIGN.md spike 5

## Context

Phase 2 requires one root `nginx.conf`, one complete file per Domain under
`domains/`, an immutable runtime manifest, and atomic replacement of an
`active` symlink. Publishing code must not be enabled until this layout is
validated against the target container image.

The production image now pins an immutable Nginx base image and runs both
Nginx and Node as the same non-root identity. The target-image rerun below
closes the provisional gate for Phase 2 publication work.

## Decision

Use the following model:

```text
runtime/
├── candidates/<deployment-id>/
│   ├── nginx.conf
│   └── domains/<domain-id>.conf
├── revisions/<deployment-id>/
└── active -> revisions/<deployment-id>
```

- `renderRootConfig()` owns the `http` context, WebSocket map, structured
  access format, and the single `include domains/*.conf` statement.
- `renderDomainConfig()` owns complete port 80/443 server blocks and accepts
  an explicit `preview` or `runtime` mode.
- `renderDomainPreview()` is a thin wrapper over the same compiler path.
- Disabled runtime projection keeps the HTTP-01 location, removes business
  locations, suppresses force-HTTPS redirect, and returns 503.
- The manifest stores source version/checksum, enabled projection,
  certificate ID, and exact generated-file checksums.
- Candidate activation uses a temporary sibling symlink followed by rename.
  A failed active validation restores the previous symlink the same way.
- The production bootstrap configuration lives in the same revision tree and
  includes the manager, default, internal-health, and user Domain servers. The
  Nginx master starts with `/data/nginx/active/nginx.conf`, so a reload follows
  the atomically replaced `active` symlink without replacing the management
  plane.
- User Domain servers use the container's non-root listeners `8080/8443`;
  Docker publishes host `80/443` to those ports.

## Host evidence

Environment:

```text
Platform: Darwin arm64
Nginx: nginx/1.27.4
Nginx executable SHA-256:
5097533017bcedef77986b5551266d108491f27e3aaeec02fb0876368665c6f3
Compiler: clang 16.0.0
OpenSSL build/runtime: 3.4.0 / 3.6.3
Modules: http_ssl, http_v2, http_v3, pcre-jit, stream, debug
```

Reproduction:

```sh
pnpm test
```

The test creates two complete candidate trees in a temporary directory and
runs the production-shaped command against both:

```sh
nginx -p <candidate>/ -t -c nginx.conf
```

It then atomically replaces `active`, validates through the symlink, injects
an unknown directive into a third candidate, confirms `nginx -t` rejects it,
switches to the broken revision to exercise post-activation detection, restores
the previous revision, and confirms the restored active tree passes again.

## Generated configuration shape

Root:

```nginx
http {
  map $http_upgrade $connection_upgrade {
    default upgrade;
    '' close;
  }
  log_format domain_manager escape=json '<structured fields>';
  include domains/*.conf;
}
```

Domain files contain a system-owned HTTP-01 location, validated Route/Header/
Advanced directives, runtime log paths derived from the normalized primary
hostname, and stable certificate placeholders in preview mode.

## Manifest evidence

The test verifies:

- root config checksum equals the generated bytes;
- domain config checksum equals the generated bytes;
- source snapshot checksum is recorded independently;
- enabled and certificate projections are explicit manifest fields;
- changing runtime inputs does not mutate the source snapshot.

## Failure injection result

| Injection | Observed result | Runtime effect |
| --- | --- | --- |
| Invalid candidate directive before activation | `nginx -t` fails | Existing `active` target unchanged |
| Invalid revision after symlink switch | Active validation fails | Previous symlink restored and revalidated |
| Disabled Domain projection | Candidate remains valid | Challenge retained; business requests return 503 |

## Target-image evidence

The spike was repeated in the locked production image on 2026-07-19:

```text
Nginx base: nginx:1.28.0-bookworm@sha256:552e7481ca93ffccd046aa658dbbed22caefbc09c66fa7cd247cbb90b8a5c609
Built image: sha256:4ae1ed9ca23980b004f94441dd4760f679559d57c9ed25163d465a9332e1172c (linux/arm64)
Runtime identity: uid=10001(nginx-manager) gid=10001(nginx-manager)
Effective capabilities: 0000000000000000
NoNewPrivs: 1
Nginx: nginx/1.28.0, OpenSSL build 3.0.15 / runtime 3.0.17
Spike filesystem: tmpfs
```

The container used a read-only root filesystem, dropped all capabilities,
started from `active -> revisions/bootstrap`, passed the internal health check,
and served the management topology on `8080/8443`. The test then atomically
switched `active` to an invalid revision, observed `nginx -t` reject the unknown
directive, restored the previous sibling symlink, and revalidated the restored
tree successfully. This also confirmed that the system-owned management and
internal-health servers remain present in every active root config.

The completed Runner was then exercised end to end in the same hardened
container: a v1 Domain draft generated a full candidate tree and manifest,
passed candidate and active `nginx -t`, atomically activated its revision,
completed graceful reload and the internal management health check, and only
then committed the Version and Domain runtime projection in SQLite. All seven
recorded Deployment steps reached `succeeded`.

## Consequences

- Config/version preview work may proceed against the shared compiler.
- The publish/reload Runner may now be enabled against this exact revision
  layout and production listener projection. Deployments on a different volume
  filesystem must repeat the atomic rename check during release qualification.
- No API introduced by the provisional spike may reload the host Nginx.
