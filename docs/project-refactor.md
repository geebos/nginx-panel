# 全项目渐进式代码优化

在不改变现有外部行为的前提下，持续改善项目设计、可维护性和测试质量。

## 设计与代码规范

* 遵循项目现有语言、框架、命名、格式和架构约定。
* 以 SOLID 为主要设计参考，优先单一职责、高内聚、低耦合。
* 优先处理重复代码、过长函数、复杂分支、隐藏依赖、魔法值和职责混乱。
* 避免过度抽象，不为尚未出现的需求设计。
* 保持公共 API、数据格式和现有兼容性。
* 不为了套用设计原则而修改简单且合理的代码。

## 初始化

首次运行时：

1. 使用 pi-lens 分析项目结构、符号关系、诊断信息和主要复杂区域。
2. 建立按收益、风险和修改范围排序的候选优化清单。
3. 优先选择高收益、低风险、可通过测试验证的小改动。

## 每轮流程

每轮只处理一个概念明确的小问题。

1. 使用 pi-lens 确认问题、引用关系、影响范围和现有测试。
2. 提出一个最小可行优化方案。
3. 调用 `design-challenger` subagent 进行执行前对抗式审查。
4. 根据审查结论处理：

   * `APPROVE`：继续执行。
   * `REVISE`：缩小或简化方案后重新审查。
   * `REJECT`：放弃该项并选择下一个候选问题。
5. 不确定现有行为时，先补 characterization test。
6. 按 TDD 执行：

   * 先新增或修改测试，并确认测试因预期原因失败。
   * 编写最少代码使测试通过。
   * 仅重构本轮涉及的代码。
7. 运行相关测试、类型检查、lint 和 pi-lens 诊断。
8. 调用 `code-reviewer` subagent 独立审查最终 diff 和测试。
9. 修复所有 `BLOCKING` 问题，重新验证并再次提交审查。
10. 审查通过后写入本轮记录。
11. 更新任务状态并调用 `ralph_done`。

## Design Challenger 要求

`design-challenger` 只提供意见，不修改代码。

重点审查：

* 问题是否真实存在并有代码证据。
* 修改是否值得，收益是否大于风险。
* 是否存在更简单的方案。
* 是否会引入额外耦合、抽象或兼容性风险。
* 是否符合 SOLID、高内聚和低耦合。
* 是否能够通过小范围 TDD 安全实施。

必须输出：

* `verdict`: `APPROVE`、`REVISE` 或 `REJECT`
* `risks`
* `simpler-alternative`
* `required-tests`

## Code Reviewer 要求

`code-reviewer` 只审查，不修改代码。

重点检查：

* 行为正确性和向后兼容性。
* 测试是否真正验证目标行为。
* 是否存在回归、边界遗漏或错误处理问题。
* 是否增加不必要的复杂度、抽象或耦合。
* 是否混入无关修改。
* 是否符合项目规范和 SOLID。
* diff 是否保持足够小且容易审查。

仅报告有证据的问题，并分为：

* `BLOCKING`
* `NON_BLOCKING`

没有阻塞问题时明确输出 `APPROVED`。

## 处理记录

每轮在 `docs/refator` 创建一个 Markdown 文件。

命名格式：

`NN.title.md`

规则：

* `NN` 从 `01` 开始递增，始终使用两位数字。
* 创建文件前先检查已有文件的最大序号。
* `title` 使用简短、清晰的英文 kebab-case。
* 示例：`01.extract-user-validator.md`。
* 不修改或覆盖已有记录。

每份记录包含：

* 发现的问题
* 问题证据和影响范围
* 原始设计
* 最小修改方案
* Design Challenger 结论
* TDD 过程
* 最终修改
* Code Reviewer 结论
* 测试、类型检查、lint 和 pi-lens 诊断结果
* 后续候选优化

## 限制

* 每轮只完成一个小改动。
* 不进行大规模重写。
* 不混入无关重命名、批量格式化或依赖升级。
* 不同时修复多个独立问题。
* 审查 subagent 不得修改文件或执行破坏性命令。
* 出现回归、影响范围扩大或无法验证时，停止本轮并记录原因。
* 测试、检查或审查未通过时，不得调用 `ralph_done`。
